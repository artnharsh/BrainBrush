HOW TO USE THIS IMPLEMENTATION
================================================================================

FOR DEVELOPERS:

1. ADDING NEW SOCKET EVENTS:
   Step 1: Define interface in backend/src/types/socketTypes.ts
   Step 2: Define corresponding interface in frontend/src/types/socketTypes.ts
   Step 3: Use type in socket.on() handler with explicit return type (: void or : Promise<void>)
   Step 4: Import type and use in socket event listener on frontend

   Example:
   ```typescript
   // In socketTypes.ts
   export interface NewFeaturePayload { roomCode: string; data: string; }

   // In socket handler
   socket.on("new_event", (payload: NewFeaturePayload): void => { ... })

   // On frontend listener
   const onNewEvent = (payload: NewFeaturePayload): void => { syncState(payload); }
   ```

2. ADDING NEW SERVICES:
   - Add explicit return type annotation to EVERY function
   - Use existing GameState, Room, or create new Result interface
   - Error handling: throw new Error("message") in service, catch in socket handler

   Example:
   ```typescript
   export const myService = async (): Promise<MyResult> => {
     try {
       // logic
       return result;
     } catch (error) {
       throw new Error("Specific error message");
     }
   }
   ```

3. HANDLING SOCKET ERRORS:
   Pattern already established in socket handlers:
   ```typescript
   socket.on("event", async (): Promise<void> => {
     try {
       // logic
     } catch (error) {
       const msg = error instanceof Error ? error.message : "Unknown error";
       socket.emit("error", { message: msg });
     }
   });
   ```

4. FRONTEND ERROR HANDLING:
   Once ErrorBoundary is created, wrap pages:
   ```typescript
   <ErrorBoundary>
     <GamePage />
   </ErrorBoundary>
   ```

TECHNICAL DECISIONS & RATIONALE
================================================================================

1. Socket Event Typing:
   Decision: Create separate payload interfaces per event type
   Why: Each event has different structure; one interface would be a catch-all
   Alternative: Would require `any` or `unknown` casts on handlers

2. Service Return Types:
   Decision: Explicit return types on ALL async functions
   Why: Enables type checker to validate calling code
   Alternative: Relying on return type inference loses validation benefit

3. Error Handling in Sockets:
   Decision: `error instanceof Error ? message : "Unknown"`
   Why: JavaScript allows throwing any value; this safely extracts message
   Alternative: Could use try-catch with type guards

4. Frontend/Backend Type Mirroring:
   Decision: Duplicate interfaces for frontend (not import from backend)
   Why: Frontend cannot import backend code; also decouples packages
   Alternative: Would require shared type package (more infrastructure)

5. AuthenticatedSocket User Type:
   Decision: Made user optional (?), always check before use
   Why: Authentication can fail; prevents null reference errors
   Alternative: Would require making user required (breaks auth flow)
