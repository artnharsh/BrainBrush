Redis (in this project)
What is Redis?

Redis is an in-memory key–value database.
Data is stored in RAM, so it is extremely fast (milliseconds).
Commonly used for caching, sessions, real-time data, pub/sub systems.

------------------------------------------------------------------
------------------------------------------------------------------

Why we use Redis in the Scribble Project

MongoDB is persistent but slower.
A real-time game needs very fast reads/writes.

Redis is used because:

Game state changes every second
Players join/leave rooms frequently
Scores and turns update instantly
So Redis handles temporary fast data, while MongoDB stores permanent data.

How Google OAuth Actually Works (short)

Flow:

User clicks "Login with Google"
        │
        ▼
Frontend → /auth/google
        │
        ▼
Google login page
        │
User signs in
        │
        ▼
Google sends user data → /auth/google/callback
        │
        ▼
Passport verifies user
        │
        ▼
User stored/fetched from MongoDB
        │
        ▼
Session created
Why OAuth is used here

Instead of building:

signup
login
password reset
email verification
security

You outsource authentication to Google, which is more secure and easier.

What Passport is doing

Passport handles:

Google authentication
session creation
user serialization
login state

Your job is only to store the user in DB.

How JWT Works (Short Explanation)

JWT is basically a signed identity token.

Example payload:

{
  "id": "user123",
  "iat": 171000000,
  "exp": 171060000
}

Flow:

User logs in with Google
        │
Backend creates JWT
        │
Frontend stores token
        │
Frontend sends token in request header
        │
Backend verifies token
        │
User allowed to access protected routes

Header example:

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

------------------------------------------------------------------
------------------------------------------------------------------

Why JWT is used here

Without JWT:

Every request → check session in DB

With JWT:

Token itself contains user identity
No DB lookup needed

Faster and stateless.

Google OAuth → Passport
Authorization → JWT
Game state → Redis
Data → MongoDB

How Socket Authentication Works

When frontend connects:

const socket = io("http://localhost:5000", {
  auth: {
    token: JWT_TOKEN
  }
});

Backend receives:

socket.handshake.auth.token

Then:

verify JWT
attach user id to socket
allow connection

Why JWT Validation is Critical

Without this step:

anyone could connect to your socket server
join rooms
send guesses
manipulate games

So socket auth is mandatory.

Using Redis Keys Instead of JSON Objects

You chose this structure:

room:AB12CD:host
room:AB12CD:players

Example in Redis:

room:AB12CD:host     → user123
room:AB12CD:players  → {user123,user456,user789}

Why this is good:

atomic operations
no race conditions
fast lookups
easy cleanup

Instead of storing:

room:AB12CD → JSON object

which causes race conditions.

------------------------------------------------------------------
------------------------------------------------------------------

3️⃣ Redis Set for Players
await redis.sadd(`room:${roomCode}:players`, userId);

A Redis Set guarantees:

no duplicates
atomic operations
fast membership checks

Example:

SADD room:AB12CD:players user1
SADD room:AB12CD:players user2

Even if two users join simultaneously:

both get added correctly

No overwriting.

This fixes the race condition problem.

------------------------------------------------------------------
------------------------------------------------------------------

4️⃣ ROOM_TTL
const ROOM_TTL = 3600;

Then:

await redis.expire(key, ROOM_TTL);

Purpose:

automatic room cleanup

If your server crashes or players leave without calling leave_room:

Redis will delete the room after 1 hour

Without TTL:

rooms stay in RAM forever
→ memory leak
→ Redis crash

------------------------------------------------------------------
------------------------------------------------------------------

5️⃣ Why exists() in joinRoom
const exists = await redis.exists(`room:${roomCode}:players`);

Purpose:

prevent joining invalid rooms

Example:

join_room ABC123

If the room doesn't exist:

throw error

Without this check:

------------------------------------------------------------------
------------------------------------------------------------------

players could create rooms accidentally
6️⃣ Why smembers()
const players = await redis.smembers(`room:${roomCode}:players`);

Purpose:

get the current player list

This list is then broadcast:

io.to(roomCode).emit("player_list", players)

So all clients update the lobby UI.

------------------------------------------------------------------
------------------------------------------------------------------

7️⃣ Why We Delete the Room When Empty
if (players.length === 0) {
  await redis.del(`room:${roomCode}:players`);
  await redis.del(`room:${roomCode}:host`);
}

Purpose:

prevent ghost rooms

Without this:

room exists but players = []

Which causes problems like:

joining dead rooms
UI confusion
memory waste

So the room is deleted immediately when the last player leaves.

------------------------------------------------------------------
------------------------------------------------------------------

8️⃣ Why Return Objects

Example:

return { roomCode, players: [userId] };

Instead of:

return roomCode

Because your socket layer likely does:

const { roomCode } = await createRoom(userId);

or

const { players } = await joinRoom(roomCode, userId);

So returning objects keeps the socket code clean.
