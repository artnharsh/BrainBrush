Redis (in this project)
What is Redis?

Redis is an in-memory key–value database.
Data is stored in RAM, so it is extremely fast (milliseconds).
Commonly used for caching, sessions, real-time data, pub/sub systems.

------------------------------------------------------------------
------------------------------------------------------------------

Why we use Redis in the Scribble Project

MongoDB is persistent but slower.
A real-time game needs very fast reads/writes.

Redis is used because:

Game state changes every second
Players join/leave rooms frequently
Scores and turns update instantly
So Redis handles temporary fast data, while MongoDB stores permanent data.

How JWT Works (Short Explanation)

JWT is basically a signed identity token.

Example payload:

{
  "id": "user123",
  "iat": 171000000,
  "exp": 171060000
}

Flow:

User logs in with Google
        │
Backend creates JWT
        │
Frontend stores token
        │
Frontend sends token in request header
        │
Backend verifies token
        │
User allowed to access protected routes

Header example:

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

------------------------------------------------------------------
------------------------------------------------------------------

Why JWT is used here

Without JWT:

Every request → check session in DB

With JWT:

Token itself contains user identity
No DB lookup needed

Faster and stateless.

How Socket Authentication Works

When frontend connects:

  auth: {
    token: JWT_TOKEN
  }
});

Backend receives:

socket.handshake.auth.token

Then:

verify JWT

Why JWT Validation is Critical

Without this step:

anyone could connect to your socket server
join rooms
send guesses
manipulate games

So socket auth is mandatory.

Using Redis Keys Instead of JSON Objects

You chose this structure:

room:AB12CD:host
room:AB12CD:players

Example in Redis:

room:AB12CD:host     → user123
room:AB12CD:players  → {user123,user456,user789}

Why this is good:

atomic operations
no race conditions
fast lookups
easy cleanup

Instead of storing:

room:AB12CD → JSON object

which causes race conditions.

3️⃣ Redis Set for Players
await redis.sadd(`room:${roomCode}:players`, userId);

A Redis Set guarantees:

no duplicates
atomic operations
fast membership checks

Example:

SADD room:AB12CD:players user1
SADD room:AB12CD:players user2

Even if two users join simultaneously:

both get added correctly

No overwriting.

This fixes the race condition problem.

------------------------------------------------------------------
------------------------------------------------------------------

4️⃣ ROOM_TTL
const ROOM_TTL = 3600;

Then:

await redis.expire(key, ROOM_TTL);

Purpose:

automatic room cleanup

If your server crashes or players leave without calling leave_room:

Redis will delete the room after 1 hour

Without TTL:

rooms stay in RAM forever
→ memory leak
→ Redis crash

------------------------------------------------------------------
------------------------------------------------------------------

5️⃣ Why exists() in joinRoom
const exists = await redis.exists(`room:${roomCode}:players`);

Purpose:

prevent joining invalid rooms

Example:

join_room ABC123

If the room doesn't exist:

throw error

Without this check:

------------------------------------------------------------------
------------------------------------------------------------------

players could create rooms accidentally
6️⃣ Why smembers()
const players = await redis.smembers(`room:${roomCode}:players`);

Purpose:

get the current player list

This list is then broadcast:

io.to(roomCode).emit("player_list", players)

So all clients update the lobby UI.

------------------------------------------------------------------
------------------------------------------------------------------

7️⃣ Why We Delete the Room When Empty
if (players.length === 0) {
  await redis.del(`room:${roomCode}:players`);
  await redis.del(`room:${roomCode}:host`);
}

Purpose:

prevent ghost rooms

Without this:

room exists but players = []

Which causes problems like:

joining dead rooms
UI confusion
memory waste

So the room is deleted immediately when the last player leaves.

------------------------------------------------------------------
------------------------------------------------------------------

8️⃣ Why Return Objects

Example:

return { roomCode, players: [userId] };

Instead of:

return roomCode

Because your socket layer likely does:

const { roomCode } = await createRoom(userId);

or

const { players } = await joinRoom(roomCode, userId);

So returning objects keeps the socket code clean.

Why switching to Pure WebSockets??
The Core Concept: Two Different Doors
Think of your roomService.ts (which talks to Redis) as the "Brain" of your room logic.

Your frontend needs a way to talk to this Brain. In web development, there are two main "doors" you can use to get there:

The HTTP Door (REST APIs): This is what you deleted. It uses app.post('/create-room') and Controllers. It is a one-time transaction: the client knocks, asks for a room, the server gives a room code, and the door slams shut.

The WebSocket Door: This is what we are using now. The client walks in, sits down, and keeps an open, continuous radio channel with the server.

The Old Workflow (Hybrid)
Before you deleted those files, your app was doing extra, unnecessary work. Here was the flow:

Frontend sends an HTTP POST request to /api/create-room.

Route (roomRoutes.ts) catches the /create-room URL.

Controller (roomController.ts) checks the user ID and calls createRoomRedis().

Backend replies with the room code.

Then, the frontend opens a WebSocket connection and says, "Hey, put me in the room I just created."

It worked, but you were using both doors for the same task.

The New Workflow (Pure WebSockets)
By deleting the Express routes and controllers, we completely boarded up the HTTP door. Now, everything goes through the WebSocket door. Your roomSocket.ts file is literally acting as your new "Controller" and "Route" rolled into one.

Here is the exact step-by-step workflow of your app right now:

Step 1: The Persistent Connection
When the user logs in and loads the lobby, your frontend runs io("http://localhost:5000"). The client and server shake hands and open a permanent, two-way pipe.

Step 2: The "Route" (The Event)
Instead of making an HTTP fetch request to a URL, the frontend simply yells into the pipe:
socket.emit("create_room")

Step 3: The "Controller" (The Listener)
Inside your roomSocket.ts file, your backend is constantly listening to that pipe. It hears the "create_room" event.
Just like your old Express controller, it extracts the userId, does a safety check, and calls the exact same createRoomRedis(userId) function from your Service layer.

Step 4: The Database (Redis)
roomService.ts does the heavy lifting. It generates the A1B2C3 code, saves the host and players to Redis, and sets the 1-hour expiration timer. It returns the data back to the socket listener.

Step 5: The "Response" (The Broadcast)
Instead of using res.status(200).json(...) like an HTTP controller, the socket listener uses socket.emit("room_created", { roomCode }) to instantly push the result back down the pipe to the frontend.

1. How Turns Work (The Index)

We do not use fractions or math to track turns. Instead, we use an Array Index (currentPlayerIndex).

If the array is ["Alice", "Bob", "Charlie"], Alice is index 0. When her turn ends, we just do +1 to move to Bob (index 1).

When the index reaches the end of the array (length), we reset the index to 0 and add +1 to the Round.

2. How We Handle "Rage Quits" (Disconnects)

If a player closes their tab, the server removes them from the array (splice).

The Math Fix: If the player who left was before the current drawer in the list, the array just shrank. We have to shift our currentPlayerIndex back by -1 so we don't accidentally skip the next person in line.

The Drawer Quits: If the person who left was the active drawer, the server instantly forces a nextTurn() so the game doesn't get permanently stuck waiting for a ghost to draw.

------------------------------------------------------------------
------------------------------------------------------------------

Server-Authoritative Timers
Why the Server? We never trust the client (browser) to keep time. If the timer was in frontend JavaScript (setTimeout), a player could cheat by hacking the code, or the timer would pause if they minimized the tab. The server holds the absolute "stopwatch" and broadcasts the time.

The Map Object: Because a server handles many rooms at once, we use activeTimers = new Map() to link a specific roomCode to a specific interval ID.

Memory Leaks: setInterval will run forever and crash your server if you don't stop it. We always call clearInterval() when the timer hits zero or when a round ends early to prevent overlapping clocks.

The Reconnect Handoff
New Player: "Hey everyone, I just joined! Send me the canvas."

Drawer: Uses canvas.toDataURL() to compress the drawing into a text string and says to the server: "Give this string to the new guy."

Server: Forwards the string.

New Player: Uses new Image() and ctx.drawImage() to instantly paste that text string onto their screen as a full picture.

This guarantees your server memory stays perfectly clean, while giving players a seamless experience!

Inside Docker, localhost means the container itself. If your backend looks for Redis on localhost, it will look inside the backend container, find nothing, and crash.
Docker Compose automatically creates a magic internal DNS. You must change your connection strings in your code (or via the environment variables above) to use the service names:

❌ redis://localhost:6379

✅ redis://redis:6379

❌ mongodb://localhost:27017/skribbl

✅ mongodb://mongodb:27017/skribbl
