1. Zustand (Global State Management)
What it is: A tiny, incredibly fast library for managing global state in React. It is the modern alternative to Redux (which is too bulky) and React Context (which has performance issues).

The Problem it Solves (Prop Drilling): If your App holds the timer state, but your TimerDisplay component is nested 5 levels deep, you normally have to pass the timer down through 5 layers of components as a "prop". This is messy.

Why we use it instead of React Context: React Context is notorious for causing unnecessary re-renders. If you put the timer and the chatMessages in the same React Context, every time the timer ticks down by 1 second, the Chat Box re-renders! Zustand uses strict selectors, meaning the Chat Box only re-renders when a message is added, ignoring the timer completely.

How we use it in Skribbl: We created a useGameStore hook. Now, our Canvas, Chat, and Scoreboard can instantly grab the exact same currentDrawer or roomCode from anywhere in the app, and update it globally without passing a single prop.

2. Axios Interceptors
What it is: A piece of code that acts like a "checkpoint" or "toll booth" that every single HTTP request and response must pass through before reaching its destination.

The Problem it Solves: When a user logs in, the backend gives them a JWT (Token). To prove who they are, the user must send that token with every single API request. Writing headers: { Authorization: token } on 50 different API calls across your app is repetitive and error-prone.

How we use it in Skribbl: 1. Request Interceptor: We intercept every outgoing request to the backend and automatically staple the JWT token to it.
2. Response Interceptor: We intercept every incoming response from the backend. If the backend replies with a 401 Unauthorized (meaning the token expired), we automatically wipe the Zustand store and kick the user back to the Login screen, no matter what page they were on.

Day 3 Concepts
1. Protected Routes (Client-Side Security)
What it is: A wrapper component in React Router that checks if a user has permission to view a page before rendering it.

Why we use it: If you don't use protected routes, a user who isn't logged in can just type localhost:5173/lobby into their URL bar and bypass the login screen.

How it works: It checks the global state (or localStorage) for a valid token. If it exists, it returns <Outlet /> (which tells React Router to render the requested page). If it doesn't exist, it returns <Navigate to="/" /> to force them back to the login screen.

2. OAuth Flow (Delegated Authentication)
What it is: Allowing a trusted third party (Google, GitHub, Apple) to verify a user's identity so you don't have to manage their passwords.

The Architecture (Why the frontend sends a token to the backend): When the user clicks "Continue with Google," Google gives the frontend a secure token. The frontend cannot be trusted to just say "I'm authenticated!" It must send that Google token to your backend. Your backend then asks Google's servers "Is this real?", and if Google says yes, your backend generates your app's custom JWT and sends it back to the frontend.

Token Hydration: If a user refreshes a React SPA, global state (Zustand/Redux) is destroyed. You must "hydrate" the state on the initial load by parsing the JWT from localStorage.

Race Conditions in Routing: If a Route Guard evaluates global state before the app finishes hydrating, authenticated users will be falsely redirected to the login screen.

Client vs Server OAuth: Client-side OAuth (@react-oauth/google) keeps the user on the SPA without screen flashes. Server-side OAuth requires redirects and "Success Pages" to catch URL parameters.

Day 3 (Authentication & Architecture)

1. Server-Side OAuth vs. Client-Side OAuth
What we used: Server-Side OAuth (Passport.js).

How it works: The React SPA physically redirects the browser (window.location.href) to the Express backend. The backend handles the secure handshake with Google, generates a JWT, and redirects the browser back to React.

Why it's secure: The Google Client Secret never touches the frontend. The backend handles the entire cryptographic verification.

2. The Token Hand-Off (res.redirect vs res.json)
The Problem: When a backend finishes logging a user in, calling res.json(token) just renders raw text on a blank screen. A React app on a different port (5173) cannot read it.

The Solution: The backend must use res.redirect to send the user back to a specific React route (e.g., /auth/success?token=xyz). React then acts as a "catcher's mitt," extracting the token from the URL, saving it to global state, and silently moving the user to the Lobby.

3. JWT Storage: localStorage vs HttpOnly Cookies
localStorage (What we chose): Vulnerable to XSS (Cross-Site Scripting) attacks if a hacker injects malicious code. However, it completely bypasses CORS (Cross-Origin Resource Sharing) headaches across different ports, and makes it incredibly easy to attach the token to WebSocket (Socket.io) connections.

HttpOnly Cookies (The Enterprise Alternative): Immune to XSS because JavaScript cannot read them. However, they require complex CORS configurations and make establishing initial Socket.io handshakes much more difficult.

The Architect's Decision: For a low-stakes gaming SPA with WebSockets, localStorage is the pragmatic choice. For a banking app, HttpOnly cookies are mandatory.

4. React 18 Strict Mode (The Double-Mount Bug)
The Problem: In development mode, React 18 intentionally mounts, unmounts, and re-mounts components in a split second to test for memory leaks. This causes useEffect to fire twice, which can break one-time logic (like processing a token from a URL, because the first pass already deleted the token!).

The Fix: Using useRef(false) as a persistent boolean flag inside the component. If the flag is true, we immediately return and block the second render from executing the logic again

1. The "Singleton" Socket Pattern
The Problem: React is reactive. If you put const socket = io() directly inside a component, React will create a brand new, separate WebSocket connection to the server every single time that component re-renders. A dozen users could instantly spawn thousands of ghost connections and crash your Node.js server.

The Solution: We create a separate socketClient.ts file that initializes exactly one instance of the socket outside the React component tree. We export that single instance (a "Singleton") so the entire app shares the exact same connection lifeline.

2. The Authentication Handshake
How it works: We set autoConnect: false so the socket doesn't fire immediately. First, we pull the JWT from localStorage, inject it into the socket payload (socket.auth = { token }), and then manually call socket.connect().

Why it matters: The backend server uses a middleware to intercept the connection attempt, verify the JWT, and immediately attach the user's ID to that specific socket socket instance. This ensures no unauthenticated users can access the game rooms or emit drawing events.

3. Server-Driven UI (Single Source of Truth)
The Challenge: In multiplayer games, a user might experience a brief internet drop, or hit "refresh" on their browser. This completely wipes their local React state, causing them to desync from the game.

The Architecture: The Backend Server is the absolute "Single Source of Truth" (SSOT). We built a master sync_state event listener. When a user reconnects, the backend blasts them with a JSON object containing the exact current state of the game (players, current drawer, timer, word choices). The frontend blindly accepts this data and uses syncGameState in Zustand to overwrite its local state, instantly snapping the user back into sync with everyone else.

Day 5 (Lobby System Architecture)
1. State-Driven Views vs. URL-Driven Views
What we did: We combined the "Create/Join" screen and the "Waiting Room" screen into a single React component (LobbyPage.tsx).

How we did it: We used a simple conditional render based on the Zustand store: if (!roomCode) { show Gateway } else { show Waiting Room }.

Why we did it: It creates a lightning-fast, seamless user experience without triggering a full page route change.

Why not the alternative (using React Router for /lobby and /room/ABCD): If we used URL routing, a user could manually type /room/ABCD into their browser before their WebSocket has successfully connected or authenticated. By driving the UI strictly through Zustand's roomCode state, we mathematically guarantee that a user can only see the Waiting Room if the server has explicitly placed them there via a socket event.

2. Host Authority (Frontend Role-Based Access)
What we did: We restricted the "Start Game" button so only the room creator can see it.

How we did it: We compare the currently logged-in user's ID with the room's Host ID: user?.id === hostId.

Why we did it: To prevent chaos. In multiplayer games, State Transitions (moving from Lobby to Playing) must be triggered by a single, authoritative source.

Why not the alternative (just letting anyone click start): Race conditions. If two players click "Start" at the exact same millisecond, the server receives two conflicting events, which can corrupt the game timer or round counter.

⚠️ The Interview Trap: If an interviewer asks, "Is hiding the button enough?" Your answer must be: "No. Frontend hiding is just for User Experience. A hacker could still manually emit the 'start_game' socket event via the console. The backend must also verify that the socket ID requesting the game start matches the host ID in the database/memory."

3. Server-Acknowledged State vs. Optimistic UI
What we did: When a user clicks "Create Room", we set isLoading(true) and show a bouncy loading screen. We do not manually set the roomCode on the frontend.

How we did it: socket.emit("create_room") sends the request, and the frontend literally just sits and waits. It only changes when the server fires back the sync_state event.

Why we did it: Because WebSockets are asynchronous and subject to network latency.

Why not the alternative (Optimistic UI): In a standard web app (like liking a tweet), you use "Optimistic UI"—you make the heart turn red instantly before the server replies to make the app feel fast. You cannot use Optimistic UI in multiplayer matchmaking. If you optimistically guess a room creation succeeded, but the server crashes, the user is stuck in a phantom room that doesn't exist. The frontend must be a "dumb terminal" that only updates when the server gives it explicit permission.

Day 5.5 (Frontend & Backend Alignment)
1. The WebSocket "API Contract"
The Problem: Unlike REST APIs (which throw 404 or 400 HTTP errors if you hit the wrong endpoint), WebSockets fail silently. If the frontend emits "join_room" but the backend is listening for "joinRoom", the app just sits there forever.

What we did: We audited the React emitters and listeners to perfectly match the Express backend's exact string names ("room_created", "player_list").

The Senior Architect Answer: "In a production environment, I wouldn't hardcode these strings in both repositories. I would create a shared events.ts file containing a TypeScript Enum (e.g., SocketEvents.JOIN_ROOM) and import it into both the frontend and backend. This completely eliminates typo-driven bugs and silent failures."

2. Payload Serialization (Strings vs. Objects)
The Problem: The frontend was emitting an object { roomCode: "ABCD" }, but the backend was strictly expecting a primitive string (roomCode: string). When the backend tried to use the object as a string, Redis failed to find the room.

What we did: We updated the frontend to emit exactly what the backend expects: socket.emit("join_room", joinCode).

The Trade-off: Emitting primitives (strings/numbers) is cleaner for simple actions. However, emitting objects (like { roomCode, password }) is more extensible if you plan to add features later without breaking the API contract. In this case, we conformed to the backend's primitive expectation to keep the codebase moving.

3. Real-Time Error Propagation
The Problem: Standard HTTP error handling (try/catch with Axios) doesn't work for WebSockets. If a user tries to join a room that doesn't exist, the backend throws an error, but the frontend's try/catch won't see it because the socket connection didn't crash.

What we did: The backend catches the Redis error and actively emits a dedicated error event: socket.emit("error", "Room not found"). The frontend has a dedicated listener socket.on("error", ...) to catch this and alert the user.

Why it matters: This is the only way to provide a good User Experience (UX) over WebSockets. The UI must be able to gracefully reset its loading state if the server rejects a real-time action.

Day 6 (Edge Cases & Multiplexing)
1. Connection Multiplexing (The "Two Tabs" Problem)
The Interview Question: "If a user opens your multiplayer app in two separate browser tabs, and then closes Tab A, what happens to Tab B?"

The Trap: Junior developers will say, "Nothing, they stay logged in."

The Senior Answer: "If not handled correctly, Tab B will experience a 'Ghost Disconnect'. Because Socket.io tracks connections by the tab (socket.id), closing Tab A fires a disconnect event to the backend. If the backend blindly deletes the user from the database on disconnect, Tab B will suddenly be kicked out of the game despite remaining open."

The Enterprise Architecture Fix: "To solve this, the backend must implement Connection Multiplexing. Instead of mapping 1 User to 1 Socket, Redis should store a Hash mapping UserId -> Array of Socket IDs [socket_1, socket_2]. When a socket disconnects, we remove only that specific socket.id from the array. We only trigger the actual 'Leave Room' logic if the array length drops to 0 (meaning they closed every tab)."

2. The SPA "Back Button" Illusion (Auth Guarding)
The Interview Question: "A user logs into a React Single Page Application (SPA), clicks the browser's back button, and sees the Login screen again. Are they logged out? How do you fix this?"

The Answer: "No, they are still completely authenticated because their JWT is safely stored in localStorage (or an HttpOnly cookie). They are just looking at the wrong view. Because SPAs don't reload the page, the React Router simply rendered the / route."

The Fix (Route Guards): "We implement a Route Guard on the Login Component using useEffect. The moment the component mounts, it checks the global auth state. If isAuthenticated is true, it instantly forces a Maps('/lobby', { replace: true }), bouncing the user away from the login screen before they can even interact with it."

3. Explicit vs. Implicit State Teardown
The Interview Question: "Why should we build a physical 'Leave Room' button if closing the browser tab automatically triggers a disconnect event anyway?"

The Answer: "Relying strictly on TCP disconnects (Implicit Teardown) is dangerous because socket timeouts can take several seconds to register if a user's internet drops. Providing an explicit socket.emit("leave_room") button (Explicit Teardown) allows the frontend to instantly clear its own local state (zustand.resetRoom()) and immediately tell the server to clean up the Redis memory, resulting in a much faster, predictable, and bug-free user experience."

The "Hybrid" Approach (Optimistic UI + Backend Relay)
1. The Drawer's Frontend (Optimistic UI)
When the person who is drawing moves their mouse, their React frontend does not wait for the server. It instantly draws the line on their own HTML Canvas (0 millisecond latency). At the exact same time, it fires socket.emit("draw_line", coordinates) to the backend.

Why: If the drawer had to wait for the backend to approve and send the line back before they could see it on their screen, the pen would feel laggy, floaty, and terrible to use.

2. The Backend (The Bouncer & Relay)
The Express server receives the coordinates. It does a quick security check: "Is this socket ID actually the current drawer?" If yes, it immediately broadcasts those coordinates to everyone else in the room using socket.to(roomCode).emit("draw_line").

Why: The backend acts as the source of truth to prevent hacking. If we did pure "Frontend to Frontend" (Peer-to-Peer), hackers could easily inject code to draw on the canvas when it wasn't their turn.

3. The Guessers' Frontends (The Receivers)
The other players in the room have a passive React Canvas. It sits there listening for socket.on("draw_line"). When it receives the coordinates from the backend, it silently draws them on the screen.

📓 Interview Notepad: Day 7 (Canvas Architecture & Latency)
1. Optimistic UI in Multiplayer Gaming
The Problem: If a user’s local actions (like drawing a line or moving a character) must make a round-trip to the server before rendering on their screen, the user will experience "Input Lag" (latency).

The Solution (Optimistic UI): The frontend instantly renders the user's action locally, assuming the server will accept it. Simultaneously, it emits the action to the server in the background.

Application in Skribbl: The Drawer's canvas draws the ink instantly via native mouse events. It does not listen to its own socket emissions.

2. Client-Server Relay vs. Peer-to-Peer (WebRTC)
The Interview Question: "Why use a Node.js server to relay drawing coordinates instead of connecting the browsers directly using WebRTC?"

The Answer: 1. Security / RBAC: A centralized server can verify if the user emitting the draw events is actually the active drawer. Peer-to-peer trusts the clients too much.
2. Late Joiners (State Sync): If a new player joins halfway through a round, WebRTC struggles to provide the historical canvas state. With a centralized server, the backend can request a base64 image snapshot from the current drawer and hand it to the late joiner (which we built in gameSocket.ts!).
